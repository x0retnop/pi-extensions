## Handoff: pi-multi-edit TUI render audit (2026-06-02)

### What was done
Audited and fixed `pi-multi-edit` rendering in Pi TUI. The extension replaces the built-in `edit` tool with batch/patch capabilities, but visual output was buggy: duplicate blocks, orphaned `Working...` indicators, and missing mode labels (`edit:batch`, etc.).

### Confirmed root causes (from session testing)
1. **`renderShell: "default"` + override of built-in `edit`** → Pi's native fallback result renderer draws its own block alongside ours, causing **duplicate headers/blocks**.
2. **`renderResult` returning `[""]` during partial** → triggers Pi's `"⚡ Working..."` fallback or leaves ghost lines.
3. **`renderCall` not showing mode label** — `edit:batch`, `edit:multi`, `edit:patch` were invisible because earlier iterations stripped them.

### Final state of `index.ts` (not yet user-tested)
- `renderShell: "self"` — suppresses built-in fallback renderer completely.
- `renderCall` — colored header with **mode label**: `edit:batch path (count)` using `toolTitle`/`accent`/`dim`.
- `renderResult`:
  - `partial` → returns `{ render() { return []; } }` (empty array, no artifacts).
  - `success` → returns **only colored diff body** (no header repeat), using `colorizeDiff()` with `safeTruncate()` per line.
  - `error` → returns colored error text only.
- Diff output:
  - Single-file: plain unified diff (no `File:` prefix).
  - Multi-file: `File: path\n${diff}` per file so `renderResult` can colorize sections.
- `safeTruncate` guards `maxWidth <= suffix.length` to prevent TUI crash on narrow terminals.

### What is NOT confirmed
User has **not yet tested** the latest rewrite in actual Pi runtime. Next step is copy `pi-multi-edit/` to `~/.pi/agent/extensions/`, restart Pi, and run an agent turn that triggers `edit` in batch/multi/patch modes.

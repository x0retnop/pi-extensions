# Delegate — отложенная фича (TODO / на подумать)

**Статус:** placeholder. Пока не реализуем, чтобы не мешалась и не усложняла основной флоу.

---

## Зачем это обсуждалось

Основная проблема текущего `subagent` tool: агент почти никогда не делегирует задачи самостоятельно, потому что сильный основной агент (Kimi) проще сделает всё сам. Но вручную хочется иметь возможность отдать текущую задачу в «параллельную копию себя» без составления подробного ТЗ.

Идея Delegate — запустить изолированную копию основного агента с тем же системным промптом и моделью, дать краткую формулировку и получить структурированный отчёт.

---

## Варианты реализации

### Вариант A: полная копия основного агента

- Системный промпт через `ctx.getSystemPrompt()`.
- Модель через `ctx.model.id`.
- Инструменты через `ctx.getSystemPromptOptions().selectedTools`.
- Запуск дочернего `pi` с `--system-prompt <tempfile> --model <parent> --tools <parent>`.
- Дополнительно `--no-context-files --no-skills`, потому что project context уже внутри переданного промпта.

Плюсы:
- Расплывчатые формулировки интерпретируются в родном контексте.
- Качество и стиль работы совпадают с основным агентом.

Минусы:
- Нужно аккуратно сохранять/воспроизводить системный промпт.
- Дорого по токенам — та же модель, тот же большой промпт.

### Вариант B: Delegate со слабой моделью + родной промпт

- Тот же системный промпт, но модель Gemma/Flash.
- Только для локальных, хорошо изолированных задач.

Плюсы:
- Дешевле.

Минусы:
- Слабые модели могут странно интерпретировать промпт Kimi.
- Риск получить некорректные правки или бесполезный отчёт.

**Вывод:** вариант B отклонён — для Delegate нужна топовая модель.

### Вариант C: Delegate как TUI-режим без agent tool

- В `/sub-agents` TUI пункт «Delegate task».
- Пользователь выбирает режим: Edit allowed / Read-only.
- Вводит короткую формулировку.
- Запускается, результат показывается в TUI.

Плюсы:
- Полностью ручное управление, никакого автоделегирования.
- Можно сделать удобный viewer отчёта и diff.

Минусы:
- Требует доработки TUI.

## Что Delegate должен получать на вход

- Текущий системный промпт родителя.
- Текущую модель.
- Активные инструменты.
- Краткую историю последних N записей сессии (message/toolResult).
- Краткую формулировку задачи от пользователя.

## Что Delegate должен возвращать

Структурированный отчёт:

```markdown
## Что было понято задачей
## Что сделано
## Изменённые файлы
## Ключевые решения
## Риски / открытые вопросы
## Рекомендуемый следующий шаг
```

## Пост-обработка результата

Варианты:
1. Показать отчёт в TUI + предложить посмотреть `git diff`.
2. Скопировать отчёт в буфер обмена.
3. Сохранить отчёт в файл `delegate-report-<date>.md`.
4. Вставить отчёт в текущий чат как user message (чтобы основной агент продолжил).

---

## Безопасность

В системный промпт делегата нужно вшить:

- «Не делай рискованных или необратимых действий без явного указания. Если сомневаешься — опиши в отчёте, но не ломай рабочий код.»
- «Не задавай уточняющих вопросов. Действуй исходя из контекста и формулировки.»
- По умолчанию предлагать read-only режим для первичного разведывания.

- Стоит ли делать кнопку «Send to chat» для автоматической передачи отчёта основному агенту?
